# 魔女採掘 仕様書

## 概要

`魔女採掘` は、独自の軽量2Dゲームエンジン上で動く、2D見下ろし型の採掘アクション・スペルリング武器ローグライトです。

このリポジトリでは、完成版の商用ゲームではなく、ゲーム性を検証できる最小プレイ可能版と、本制作へ進めるためのエンジン土台を実装しています。

## 現在実装済みの内容

- C++20 + SDL3 + CMake のプロジェクト構成
- SDL3ウィンドウ、レンダラー、メインループ、delta time
- キーボード/マウス入力の抽象化
- プレイヤー移動とカメラ追従
- プレイヤーの壁衝突
- 32x32タイル単位のチャンク式地下マップ
- 地形種別: 空洞、土、岩、鉱石
- 配列ベースの地形管理
- プレイヤー周辺チャンクのみの生成/描画
- スコップと松明の円形のスペルリング
- スコップによる地形掘削
- 基本暗闇と、光源半径内だけを描画する円形視界
- 敵のオブジェクトプール
- 掘削で空洞化したタイルからの敵スポーン
- 壁を貫通しない敵移動
- プレイヤー周辺の空洞タイルを使ったBFS追跡
- 敵の接触ダメージ
- リングアイテムによる敵へのダメージ
- 敵撃破による経験値獲得
- レベルアップ時の3択強化
- リング中心の前方ずらし
- スペルリング全体の投げ/戻り状態
- FPS、敵数、アクティブチャンク数などのデバッグ表示
- `data` / `assets` 配下の変更検知による自動リロード
- `data/game_balance.cfg` によるランタイムバランス調整
- テスト起動用 `run_game.bat`
- コード変更を監視して自動ビルド/再起動する開発用 `dev_auto_reload.bat`

## ビルド方法

### 必要環境

- C++20対応コンパイラ
- CMake 3.22以上
- SDL3

SDL3は次のどちらかで使用できます。

- `external/SDL` にSDL3のソースを置く
- vcpkgなどで `SDL3Config.cmake` を見つけられる状態にする

### vcpkgを使う場合

```powershell
cmake -S . -B build -DCMAKE_TOOLCHAIN_FILE=C:/path/to/vcpkg/scripts/buildsystems/vcpkg.cmake
cmake --build build --config Release
.\build\Release\MajoShovel.exe
```

### `external/SDL` を使う場合

```powershell
git clone https://github.com/libsdl-org/SDL external/SDL
cmake -S . -B build-nopch -DCMAKE_DISABLE_PRECOMPILE_HEADERS=ON
cmake --build build-nopch --config Release --target MajoShovel
.\build-nopch\Release\MajoShovel.exe
```

この環境では、Visual Studio BuildTools + 日本語を含むパスでSDL3のプリコンパイル済みヘッダー生成が失敗したため、`-DCMAKE_DISABLE_PRECOMPILE_HEADERS=ON` を指定してビルド確認しました。

## 起動方法

ビルド後、プロジェクト直下の次のbatを実行します。

```powershell
.\run_game.bat
```

`run_game.bat` は以下の順に実行ファイルを探します。

- `build-nopch\Release\MajoShovel.exe`
- `build\Release\MajoShovel.exe`
- `build-nopch\Debug\MajoShovel.exe`
- `build\Debug\MajoShovel.exe`

開発中にコード変更へ自動追従したい場合は、次のbatを使います。

```powershell
.\dev_auto_reload.bat
```

`dev_auto_reload.bat` は初回ビルド後にゲームを起動し、監視対象に変更があるたびにゲームを停止、再ビルド、再起動します。

## 自動リロード

自動リロードには2種類あります。

1. ゲーム内データのランタイムリロード
2. コード変更時の開発用ビルド/再起動

### ゲーム内データのランタイムリロード

実行中、ゲームは次のディレクトリを監視します。

- `data`
- `assets`

監視対象内のファイルが追加、削除、更新された場合、ゲームは変更を検知して自動リロードします。

現在のリロード内容:

- `data/game_balance.cfg` を再読み込み
- 読み込み成功時にゲーム状態をソフトリセット
- プレイヤー、地形、敵、スペルリング、レベル状態を初期状態から再生成
- 画面上に短時間 `RELOADED` 表示

読み込みに失敗した場合は、現在のバランス値を維持し、画面上に `DATA LOAD FAILED` を表示します。

### ランタイム調整できる値

`data/game_balance.cfg` では、次のような値を実行中に変更できます。

- プレイヤー速度、プレイヤー半径
- 光源半径
- リング半径、リング速度
- リングずらし距離
- リング投げのクールダウン、速度、距離、最大時間、戻り速度
- 土、岩、鉱石のHP
- 敵の速度、半径、HP、経験値
- 掘削何タイルごとに敵を出すか
- 敵のソフト上限数
- レベルアップに必要な経験値係数

例:

```text
player_speed=180
light_radius=200
spell_ring_radius=72
enemy_speed=70
```

保存すると、実行中のゲームが自動で再読み込みします。

### コード変更時の開発用リロード

`dev_auto_reload.bat` を使うと、C++コード変更時に外部ウォッチャーが自動で再ビルドし、ゲームを再起動します。

監視対象:

- `src`
- `data`
- `assets`
- `CMakeLists.txt`

処理手順:

1. 変更を検知
2. 約0.7秒待って連続保存をまとめる
3. 実行中の `MajoShovel.exe` を停止
4. `cmake -S . -B build-nopch -DCMAKE_DISABLE_PRECOMPILE_HEADERS=ON` を実行
5. `cmake --build build-nopch --config Release --target MajoShovel` を実行
6. ビルド成功時にゲームを再起動
7. ビルド失敗時はゲームを再起動せず、次の保存を待つ

これはC++バイナリをプロセス内で差し替えるホットリロードではありません。現在の段階では、安全で単純な「再ビルドして再起動」方式を採用しています。

## 操作方法

- `WASD` / 矢印キー: 移動
- マウス位置: 向き
- `Space`: 押している間、リング中心を前方へずらす
- `J` / 左クリック: 現在のスペルリング全体を投げる
- `1` / `2` / `3`: レベルアップ時の強化選択
- `F1`: デバッグ表示の切り替え
- `Esc`: 終了

## ゲーム仕様

### プレイヤー

- 2D見下ろし画面上を移動します。
- 地形タイルの壁は通り抜けられません。
- 壁に当たった場合は、軸ごとに移動を解決して壁沿いに滑ります。
- マウス方向を向きます。
- 周囲にスコップと松明を装備したスペルリングを持ちます。
- 敵を倒すと経験値を得ます。
- 経験値が一定値に達するとレベルアップします。

### スペルリング

- 現在は1本のスペルリングのみを実装しています。
- リング上にはスコップと松明が装着されています。
- 初期配置では、スコップと松明は同じリング上で180度ずれた位置にあります。
- 初期リング半径は1マス分、つまり16pxです。
- アイテム位置は物理演算ではなく、角度、半径、速度から数式で計算します。

計算式:

```text
itemAngle = baseAngle + itemIndex * spacing
x = ringCenter.x + cos(itemAngle) * radius
y = ringCenter.y + sin(itemAngle) * radius
```

### リング中心ずらし

- `Space` を押している間、リング中心がプレイヤーの向き方向へ移動します。
- 離すと滑らかに通常位置へ戻ります。

### リング投げ

- `J` または左クリックで発動します。
- リング中心ごと前方へ飛びます。
- 一定距離または一定時間が経過すると戻り状態になります。
- 戻り状態ではプレイヤーへ向かって移動し、近づくと通常状態に戻ります。
- クールダウンがあります。

### 暗闇と光源

- ゲーム画面は基本的に暗闇です。
- 光源から半径 `light_radius` px の円内だけ、地形、敵、リングアイテムを描画します。
- 初期設定では `light_radius=200` です。
- 光は半透明グラデーションではなく、円の内側は見える、外側は見えないというハードな視界です。
- 現在の光源はプレイヤー位置と松明位置です。
- 松明はリング上を移動するため、プレイヤー周囲とは別の場所を照らせます。

### 地形

- タイルサイズは16pxです。
- チャンクサイズは32x32タイルです。
- 地形はチャンク内の配列で管理します。
- 土、岩、鉱石にはHPがあり、スコップの当たり判定でダメージを受けます。
- HPが0になると空洞になります。

### 敵

- 敵は固定容量のオブジェクトプールで管理します。
- スコップで地形を掘り、タイルが空洞化したとき、その掘削地点からスポーンします。
- 開始直後から時間経過だけで湧く定期スポーンは行いません。
- プレイヤー周辺のタイルグリッドにBFSで距離場を作り、空洞をたどってプレイヤーへ近づきます。
- 敵は地形の実タイルを参照して移動し、壁を貫通しません。
- プレイヤーに接触するとダメージを与えます。
- スコップまたは松明に当たるとダメージを受けます。
- HPが0になると非アクティブ化され、経験値を落とします。

### レベルアップ

レベルアップ時はゲーム進行が一時停止し、次の3択を表示します。

1. リング半径 +10%
2. リング速度 +10%
3. スコップ威力 +1

## アーキテクチャ

### `src/engine`

ゲーム固有の軽量エンジン層です。汎用エンジン化は狙わず、このゲームに必要な機能だけを持たせています。

- `App`: SDL初期化、ウィンドウ生成、メインループ、終了処理
- `Time`: delta time、FPS計測
- `Input`: キーボード/マウス入力の抽象化
- `Renderer`: SDL描画APIのラッパー
- `Camera`: ワールド座標とスクリーン座標の変換
- `ObjectPool`: 敵や将来のエフェクト用の固定容量プール
- `FileWatcher`: `data` / `assets` 配下の変更監視

### `tools`

- `dev_auto_reload.ps1`: コード変更を監視し、自動ビルドとゲーム再起動を行う開発用スクリプト

### `src/game`

ゲーム固有ロジックです。

- `Game`: 各システムの更新/描画順を管理
- `Player`: プレイヤー状態、移動、向き
- `TileMap` / `Chunk`: チャンク式タイルマップ
- `SpellRingSystem` / `SpellRingItem`: スペルリング状態、リングアイテム位置
- `DiggingSystem`: スコップによるタイル掘削
- `EnemySystem`: 敵のスポーン、更新、戦闘、プール管理
- `LevelSystem` / `UpgradeSystem`: 経験値、レベルアップ、強化選択
- `DebugOverlay`: デバッグ表示

### `src/data`

- `GameBalance.hpp`: 現在のバランス値を集約しています。
- `RuntimeBalance.hpp` / `RuntimeBalance.cpp`: `data/game_balance.cfg` を読み込むランタイムバランス定義です。

将来的にはJSON、TSV、またはスプレッドシート由来のデータに移行しやすいよう、数値をゲームロジックから分離しています。

## パフォーマンス方針

- 土タイル1個ごとの重いオブジェクトは作りません。
- 地形はチャンク内の配列で管理します。
- プレイヤー周辺チャンクのみを生成/描画します。
- 掘削判定はスコップのhitRadiusに重なるタイル範囲だけを見ます。
- 敵は毎回 `new/delete` せず、固定容量プールで管理します。
- スペルリング武器はRigidbody的な処理ではなく、数式ベースで制御します。
- 当たり判定は軽量な円/矩形/グリッド判定を使います。
- ファイル監視はOS依存APIではなく、軽量なmtimeポーリングで実装しています。
- 描画は現状SDLの即時描画ですが、タイル数が増えた場合はチャンク描画キャッシュまたはバッチ化を優先します。

## 現在の制限

- プレイヤーと地形の衝突はまだ実装していません。
- 敵は壁を貫通しませんが、追跡はプレイヤー周辺のBFS距離場に基づく簡易実装です。
- 照明は本格的なライトマップではなく、光源円内/外の描画切り替えです。
- 画像素材、音、セーブ、ゲームパッド入力は未実装です。
- 自動テストターゲットはまだありません。
- デバッグフォントは内蔵の簡易ASCII表示です。
- C++コード変更のホットリロードは未対応です。コード変更後は再ビルドが必要です。
- `dev_auto_reload.bat` 使用時は、コード変更後の再ビルドとプロセス再起動を自動化できます。
- 実行中プロセス内でのC++関数差し替え、DLLホットスワップ、状態を保ったままのコード更新は未対応です。

## 仮決定

- 最初のプレイ可能版では、手作業マップではなく簡易プロシージャル地形を使います。
- ワールド原点周辺は、開始直後に移動できるよう空洞にします。
- SDL_image、SDL_ttfはまだ使いません。
- 見た目は矩形、円、線、簡易ドット文字で代用します。
- 敵の追跡はBFS距離場で実装し、必要になったら群れ制御や詰まり回避を追加します。
- プレイヤーの地形衝突は、洞窟移動ルールを決めた後に追加します。
- 変更監視はまずポータブルなポーリング実装にし、必要になった段階でWindowsのファイル通知APIなどへ置き換えます。
- コード変更の自動反映は、まず外部ウォッチャーによる再ビルド/再起動方式にします。

1. プレイヤーと地形の衝突

3. ゲームパッド入力と入力リマップ
4. 掘削、敵撃破、リング投げのエフェクトプール
5. バランス値の外部データ化
6. リロード対象を画像、音、ステージ定義へ拡張

8. 捕獲アミ、捕獲敵のリングアイテム化

10. 数学、チャンク参照、オブジェクトプール、レベルアップ処理の簡易テスト
